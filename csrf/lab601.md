## Title
 Server-Side Request Forgery (SSRF) at PulsePay

## Vulnerability Type
Server-Side Request Forgery (SSRF)

## Summary  
In http://kzlabs.in/601.php.  website fetches the URL from user input which is Vulnerable to SSRF. which allows attacker to make the server send requests to internal systems or external services, leading to information disclosure or further attacks .
## Vulnerable Endpoint
http://kzlabs.in/601.php

## Steps to Reproduce  
1. Navigate to the following URL : `http://kzlabs.in/601.php`
2. Go to webHook Section.
3. In `Send Test Event`, You will see the Input box for `Webhook Url Endpoint`
4. As i enter thhe   `File:///etc/passwd/` You see the server returns the Sensitive file’s contents.  


## Payload Used
`File:///etc/passwd`

## Impact
 It allows attackers to access and scan internal systems, mapping infrastructure behind the firewall.
 It allows attackers to read sensitive local files (e.g., /etc/passwd) and retrieve cloud metadata to steal credentials.
 It allows attackers to interact with internal APIs and services, potentially leading to remote code execution.
  
## Recommendations for fix

 Validate and sanitize the redirectUrl parameter to ensure that it does not contain any malicious content. This can be done by:

1. Filter out HTML tags like: `<script>`, `<img>`, `<svg>` from the Report Name field before saving anything to the database
2. Filter out JavaScript methods like: `alert()`, `confirm()`, `prompt()` so even if a tag slips through the method won't execute
4. If you're using PHP then use `htmlspecialchars()` function before rendering any user input back to the page
5. Use Cloudflare as they have so many WAF rules that almost all XSS payloads will be blocked automatically before even reaching the server
